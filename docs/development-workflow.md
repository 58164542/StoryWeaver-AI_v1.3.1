# Development Workflow

## 目的

这份文档负责说明本仓库的开发默认动作：

- 哪条分支是稳定线
- 日常开发从哪里切分支
- 什么情况下要开 worktree
- 出问题时先查什么、怎么回滚

## 分支角色

### `main`

- 默认稳定线
- 只承接已经验证过或准备稳定保留的内容

### `codex/team-dev`

- 当前共享集成线
- 日常开发默认先同步这条分支，再切任务分支

### 任务分支

- 中等及以上任务默认新开分支
- 建议使用清晰前缀，例如 `feat/`、`fix/`、`docs/`、`chore/`、`hotfix/`

## 什么时候新开分支

### 小改动

满足大部分条件即可视为小改动：

- 只改 1 到 2 个文件
- 只涉及轻量文案、注释、格式或局部修正
- 不改变运行方式、数据结构或对外行为

### 中等及以上任务

出现以下任一情况，就按中等及以上任务处理：

- 改 3 个及以上文件
- 改脚本、配置、结构、流程
- 需要以后能清晰回滚
- 需要额外验证运行时行为

这类任务默认从 `codex/team-dev` 同步后再切任务分支。

## worktree 规则

`worktree` 用来隔离工作现场，不用来替代分支。

### 默认不需要

如果当前只做一条任务线，而且当前目录没有需要长期保留的现场，通常只开分支就够了。

### 建议使用的场景

- 正在做 feature，但临时要 hotfix
- 需要同时保留稳定现场和开发现场
- 两条任务线需要并行推进，不想来回 stash / checkout

### 重要提醒

Codex 或其他 runtime 的不同 session / thread 只隔离上下文，不会自动隔离 Git 工作目录。

## 开工前默认声明

中等及以上任务开始前，默认先给出这四项判断：

1. 当前任务主题或工作线
2. Git 动作：当前分支继续，还是新开分支
3. 是否需要 worktree
4. 影响范围：局部、共享层，还是会影响多个模块

如果任务触达共享层、运行脚本、配置入口或多模块边界，这四项应在开始改动前先明确。

## hotfix 与回滚

### hotfix

- 紧急修复默认从 `main` 切 `hotfix/*` 分支
- 修好后再按项目分支规则回灌到 `codex/team-dev`

### 回滚前先查什么

优先检查：

- 是不是服务、脚本或配置没按预期工作
- 是不是环境或数据变化导致
- 是不是最近改动确实引入了可定位回归

只有在最近改动边界清晰且存在“最后已知可用版本”时，才优先回滚。

## handoff 规则

每次较大任务结束前，尽量更新：

- `docs/handoff/current.md`
- `docs/handoff/next.md`
